# 仕様書テンプレート（最小・指示書形式）

各機能をこの4セクションだけで記述する。フォーマットを固定することで、どのエージェントが書いても品質が安定する。
`plan`（どう作るか）とは役割を分け、ここには **何を満たすか（I/O 契約と振る舞い）** だけを書く。

## いつ・どこまで書くか（2層で育てる）

spec.md は一気に全部書かず、**粒度を2層に分けて段階的に育てる**。理由は、関数レベルの I/O は実質テストの契約であり、実装のずっと前に全 Phase 分を書くと陳腐化し、TDD の red テストと二重管理になるため。

| いつ | 何を書くか | 粒度 |
|------|-----------|------|
| **Phase 0（ドメイン設計時）** | システム全体の入出力 ＋ **Phase をまたぐモジュール間インターフェース（境界の契約）** ＋ 各機能の **目的 / 完了の目安** | 粗い（契約レベル） |
| **各 Phase の着手時（plan mode）** | その Phase で実装する **関数レベルの入力・出力 / 振る舞い** | 細かい（実装目標レベル） |

> **境界＝早く固める / 内部の関数＝直前に固める。** 境界の契約を Phase 0 で確定しておくと Phase 同士が統合時に噛み合い、関数レベルは just-in-time なので正確で陳腐化しない。各機能の関数レベル詳細は、その Phase の着手ゲートで red テストを書く直前に追記する。
