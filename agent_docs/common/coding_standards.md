# コーディング規約（スタック非依存の原則）

CLAUDE.md「品質維持・技術的負債対策」の詳細版。**どの言語・フレームワークでも共通する原則**をまとめる。
具体的なビルド/テスト/lint コマンドや言語固有の規約は、使用スタックのガイド（`agent_docs/stacks/<該当>.md`、例: ROS 2 / C++ は `ros2_cpp.md`）に従う。

## 設計方針

- **ロジックをフレームワークから独立させ、単体テスト可能にする。** 入出力が明確な純粋関数／クラスとして実装し、フレームワーク側（ノード／ハンドラ／コントローラ等）はその呼び出しに徹する。
- マジックナンバー禁止。閾値・設定値は設定パラメータか名前付き定数にする。
- **1 つの責務＝1 つのモジュール**（ファイル／クラス）にする。

## テスト（TDD）

- 機能ごとに代表ケース＋境界ケースをテストする。
- **レッドはアサーション失敗で示す。コンパイル／ビルドエラーはレッドと認めない**（CLAUDE.md のフロー参照）。
- テストは決定的に。実時間・乱数・外部 I/O・実機に依存しない固定フィクスチャを使う。
- 数値比較は許容誤差付きで行う（言語の対応する手段を使う）。

## 自動品質ゲート

- format / lint / 静的解析を導入し、テストと同じコマンドで実行できるようにする（具体ツールはスタックガイド）。
- 警告ゼロを維持。抑制する場合は理由をコメントで明記。

## コメント・ドキュメント

- public API に説明コメント。
- 「なぜ」を書く。「何を」はコードで表現する。仕様書・計画ドキュメントの設計意図を該当箇所に短く参照する。
- `TODO` / `FIXME` には理由と対応条件を併記。放置 TODO を負債化させない。

## コミット・PR

- 作業ブランチで開発し、メインへは PR 経由のみ（メイン直接変更禁止）。
- PR は小さく、原則 1 機能（または 1 つの垂直スライス）単位。
- PR 説明に「対応する計画フェーズ／機能」「テスト内容」「DoD チェック結果」を記載。
