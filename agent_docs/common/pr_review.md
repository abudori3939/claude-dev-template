# PR・レビュー運用（スタック非依存）

ブランチ → PR → レビュー → マージの小さなフローを安定して回すための運用詳細。
CLAUDE.md「ブランチ・PR・レビュー」のゲートを満たしたうえで、PR はここに従って進める。

## 基本フロー

1. 作業ブランチ上で実装する（ブランチの切り方は CLAUDE.md「セッション開始時の判定 / 手順0」）。
2. 実装が完了したら、**コードレビューに出す前にユーザーの承認を得る**。実装内容のサマリを提示し、一度報告・停止する。
3. ユーザー承認後、PR を作成する。
4. **コードレビューエージェント（実装者とは別エージェント）**がレビューし、**GitHub の PR 上にコメント**を返す（レビューを GitHub 上で管理する）。
5. ユーザーもレビューに参加し、疑問が解消されるまで PR 上でやり取りを続ける。
6. CI が緑で、ユーザーが問題ないと確認したらマージする。

## 役割分担

- 実装エージェントとコードレビューエージェントは**必ず別**にする（自分の実装を自分でレビューしない）。
- 機械的な指摘（format / lint / 静的解析）は自動ゲートに任せ、レビューは**設計判断**に集中する。

## 指摘への対応（重要）

PR でレビュー指摘を受けて修正したあとは、**対応状況を必ず「追加コメント」で報告する。**

- **最初の（説明）コメントを編集で上書きして最新化してはならない。** 上書きすると、どの指摘にどう対応したか／対応しなかったかの履歴が消え、人間が追跡できなくなる。
- 修正後は**新しいコメントを追加**して報告する。指摘ごとに「**対応した**（どう直したか）／**対応しない**（その理由）」が分かるように書く。
- 指摘がレビューコメントのスレッドに紐づく場合は、**そのスレッドに返信する形**で対応を書くと、指摘と対応の対応関係が明確になる。

## マージ条件

- CI（build + test + lint）が緑。
- レビュー指摘が解消され、ユーザーが承認した。
- 「PR で合格したコードのみマージ」は人手ではなく**緑チェックで保証**する（CI が緑でない PR はマージしない）。
